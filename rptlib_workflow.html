Submission Workflow

MAPIS TRH에 실제 적용할 결재 플로우. 결재·분류 확정·번호 발번·인덱싱·접근권 부여·알림이 앱 안에서 자동으로 도므로, 사람이 직접 개입하는 단계는 「작성자 제출」과 「리뷰어 결정」 둘뿐입니다.

1MAPIS TRH의 역할 — 사람 2 + 시스템

Author

작성자 (연구원)

누구나. 표준 템플릿으로 보고서를 쓰고, 분류 질문에 답하고, 파일을 붙여 제출.

Reviewer

리뷰어 (라이브러리 책임자)

제출물 검토 · 분류 최종 확정 · 승인/보완/반려. 더불어 인덱스·접근권·감사 같은 거버넌스도 담당.

ZyLab의 Approver + ZyLab Admin + Tech CoE를 하나로 통합 — 같은 신뢰 역할이라 분리할 이유가 없음. (수출통제는 필요 시 GT&C에 인앱 코멘트로 협의)
System

시스템 (자동)

번호 발번 · 표지 기입 · 인덱스 게시 · 등급별 접근권 설정 · 3채널 알림. 사람이 아님 — ZyLab Admin·IT가 손으로 하던 사무처리를 흡수.

2제출 플로우 — 5단계

Author
1. 작성 & 분류
표준 템플릿 작성 → 메타 입력 → 분류 결정트리로 등급 제안 → 파일 첨부 → 제출
›
System
2. 라우팅 & 알림
리뷰어에게 결재 요청 전달
AUTO · 이메일 + To-Do
›
Reviewer
3. 검토 & 결정
보고서·분류 확인 후 승인 / 보완요청 / 반려. 한 화면(Review)에서 처리
›
System
4. 발행
순차 번호 발번 · 표지 기입 · 해당 지역 인덱스 게시 · 등급별 접근권 설정 · 알림
AUTO · 승인 즉시
›
System
5. 라이브
즉시 검색 가능(대기시간 없음). 등급에 따라 접근 통제(View only ↔ Read/Download/Print)
✕
반려 — ITAR / 수출제한(EAR 99 제외) : 게시하지 않고 중단(Stop). 통제 저장소로 라우팅.
↩
보완요청 : 리뷰어 코멘트와 함께 작성자에게 반송(알림) → 수정 후 재제출.
4
Class 4 경로 : 접근 가능한 명단(named individuals)을 인앱에서 지정 → 접근권 자동 부여. 신규 보안그룹이 꼭 필요할 때만 IT 요청이 자동 생성.

3핸드오프마다 — 3채널 알림 자동

MAPIS 토스트
화면 내 즉시 피드백 — "제출됨", "승인됨" 등.
이메일
담당자(리뷰어·작성자)에게 자동 발송 — 메일 수동 작성 불필요.
To-Do 큐
리뷰어의 결재 큐 / 작성자의 보완 큐에 할 일로 적재 — 놓치지 않게.

4참고 — 왜 이렇게 짧아지나 (ZyLab 대비)

ZyLab 6개 역할(Author·Approver·Admin·Tech CoE·GT&C·IT) · 메일/오프라인
MAPIS 2개 역할(Author·Reviewer) + System 자동
번호: admin에 요청→수령 / 표지 수동 기입
승인 시 자동 발번 + 자동 기입
게시: PDF 전송→admin이 포털 로그인·수동 업로드 / 72시간 이내
제출 시 업로드 완료→승인 즉시 자동 게시
Class 4: IT 티켓→AD그룹 생성→수동 할당
인앱 명단 지정→접근권 자동
설계안 · 프로토타입 시각화용. 이 플로우는 Upload 화면의 상태 타임라인(작성자 시점)과 Reviewer 화면의 Review queue(리뷰어 시점)로 구현됩니다.
