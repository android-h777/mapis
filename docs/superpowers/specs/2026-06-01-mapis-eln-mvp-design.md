# MAPIS ELN — MVP 프로토타입 설계

> 작성일: 2026-06-01 · 단계: 설계 확정(브레인스토밍 산출물) · 다음: 구현 계획(writing-plans)

---

## 0. 이 문서의 성격과 프로젝트 목표

**산출물 = 연구원에게 보여줄 피드백 수집용 프로토타입.** 완제품/프로덕션 구현이 아니다.

- **시각적 설득력이 1순위.** 연구원은 DB·데이터 모델이 아니라 *화면*에 반응한다. "오, 이거 좋다"가 화면에서 보여야 한다.
- **엔진성 기능(B 자동묶기·D 계보)은 목업 데이터로 "그럴듯하게"** 보여주면 충분하다. 실제 자동분류/계보 엔진을 구현할 필요 없음 — *그래 보이게* 만들어 피드백을 끌어낸다.
- **완벽한 합의보다 "보여줄 만한 한 장"** 이 우선. 모든 설계 결정은 결국 *화면에 어떻게 그려지나*로 수렴한다.
- **구현 토대:** `/mnt/f/mpm_project/mapis_lib/`(실제 mapis 프로덕션 CSS/JS, SUIT 폰트) + `body.popup.type2` 셸 위 **정적 HTML/CSS/JS**. 출발 파일 참고: `apis_eln_pop.html`.
- 시스템명 표기: 항상 대문자 **MAPIS**.

## 1. 북극성(Vision)

- ❌ 어쩔 수 없이 시켜서 하는 **숙제 같은 프로그램** (= 대부분의 ELN / BIOVIA의 실패 지점)
- ✅ 연구원이 먼저 **"쓰고 싶은" 도구** (intrinsic pull)
- 최고의 차별점: **SharePoint에 개별 파일로 관리하는 것보다 확실히 나은** 무언가.
  - 파일 = 죽은 데이터(저장만). MAPIS ELN = 살아있는·연결된·되짚어지는 데이터.

**설계 판단 기준:** 기능을 더하기 전 항상 "이게 연구원이 *쓰고 싶게* 만드는가? SharePoint 파일보다 나은가?"로 묻는다.

## 2. 경계 (불변 — 드리프트 금지)

- **분석은 formulator에 일임** — 배합 물성·원가·변수↔물성·배합 비교는 전부 formulator의 일. ELN에 분석을 넣지 않는다.
- **ELN의 일 = 기록 + 정리 + 검색 + 연결 + 회상.** (분석 아님)
- ELN은 formulator 배합/결과를 *참조·임베드·열기 링크*로만 연계한다.
- ※ 과거 2회 formulator 기능(배합분석)으로 드리프트한 이력 있음 — 이 경계는 모든 결정에서 재확인한다.

## 3. MVP 심장 = A + B + D

선별 기준: **"SharePoint가 구조적으로 못 하는데 ELN만 할 수 있는가."**

| 축 | 내용 | 위치 |
|---|---|---|
| **A. 기록** | 저마찰 기록 — 빈 페이지 없는 시작, 카테고리 골격 | 심장(전제조건) |
| **B. 정리** | 저절로 정돈 — 자동 묶기, 기간별 타임라인, 개인 홈 | 심장(차별 강함) |
| **D. 연결** | 살아있는 데이터 — 참조 그래프, 계보, 모듈 라이브링크 | 심장(정체성) |

- **A**가 약하면 B·D가 정리할 데이터(연료)가 안 쌓인다 → 반드시 포함하되 *마찰 제거*로만.
- **B·D는 같은 엔진의 양면.** "노트가 어떻게 연결되나(D)"를 알면 "무엇을 묶을지(B)"가 거의 공짜로 나온다. 하나의 관계 그래프로 설계.
- **C(검색)** = D 그래프의 부산물 → 후순위. **E(공유)** = SharePoint 강세 영역 → 후순위(노트북 단위만 최소 포함).

## 4. 노트 속성 (B·D의 연료)

### 4.1 status — 생애주기(시간 따라 변함)
- 값: `진행중`(기본) · `성공` · `실패` · `보류`
  - (`실패`/`중단` 통합 결정 — 실패 하나로)
- **조작:** 드롭다운 아님, **탭/세그먼트 토글 원클릭**.
  - 상세 화면: 제목 근처 접근 쉬운 위치.
  - 리스트 화면: 행에서 인라인으로 바로 변경.
  - 상세·리스트가 **동일 컴포넌트**를 공유(일관성).
- **결재(승인 워크플로우) 없음** (이번 MVP 제외).
- **색 매핑(계보·리스트 시각화용):** 진행중=파랑 · 성공=초록 · 실패=빨강/회색 · 보류=노랑.
- status는 화면 모드도 좌우한다 → 7.2 참고(진행중=쓰기 모드 / 결말=연결 모드).

### 4.2 카테고리 — 종류(거의 안 변함)
- 노트의 *틀*. 생성 시 1회 선택. 선택하면 해당 **골격(섹션 구성, A2 템플릿)** 이 깔림 → A1(빈 페이지 없는 시작)과 연결.
- **MVP 구현:** `배합실험`(Formulation — 골격 제대로) + `일반/메모`(General — 골격 없음).
- **확장 슬롯:** `합성/반응`·`분석/측정` 등은 *카테고리 값만 추가하면 되는 구조*로 열어두되, MVP에선 골격 = 일반과 동일(점진 확장). typed-experiment 모델 계승.

### 4.3 자유태그 — 자유 키워드(참조로 표현 안 되는 것만)
- **완전 수동 입력, LLM 없음, 가볍게.** (자동 제안 안 함)
- 단 **정규화 저장** — 노트에 박힌 문자열이 아니라 별도 태그 엔티티 + `note↔tag` 참조. → 태그로 노트 묶기·검색 가능(살아있는 데이터).
- **최근 사용 태그를 사용순 등으로 모아보기** → 기존 어휘 재사용 유도(LLM 없이 어휘 수렴, 입력 마찰 감소).
- 강요하지 않음 — 안 써도 시스템(B·D)은 참조·status·카테고리로 정상 작동.

### 4.4 참조(FK) — D 그래프의 핵심 (태그와 별개, 중복 금지)
- 이미 key로 연결된 것: **원료**(Rawmathub) · **배합**(formulator) · **프로젝트**(NPDI) · **다른 노트**(`[[링크]]`).
- 이것들은 태그로 복제하지 않는다(중복 관리 방지). B(묶기)·검색은 이 참조 관계를 *쿼리*해서 만든다.
- 노트 1개가 **여러 프로젝트**를 동시에 가리킬 수 있음(참조라서 N:N).

## 5. 계층 / 공유 모델

### 5.1 계층 — 2단
```
노트북 (Notebook)          ← 사이드바의 묶음 단위
  └─ 노트 (Note)            ← 개별 실험·기록
        ├─ 참조: 프로젝트·원료·배합·노트링크 (N:N)
        ├─ status / 카테고리 / 자유태그
```
- **프로젝트는 컨테이너가 아니라 참조** — 노트가 N개 프로젝트를 가리킴. 한 노트북이 여러 프로젝트에 걸쳐도 됨.
- 사이드바: 노트북 트리(2단) + `내 것` / `공유됨` 구분.

### 5.2 공유 — 노트북 단위
- **노트북을 팀과 공유** → 그 안 노트가 권한 상속(노트 단위 개별 설정 없음).
- 프로토타입 화면: "내 노트북 vs 공유된 노트북" + 노트 위 코멘트(E, 최소 형태).

## 6. 화면 전체 지도 (6화면)

```
[1] 개인 홈/대시보드   ← 진입점, "내 연구 한눈에" (B: 자동배치·자동묶기·타임라인)
      │  (노트북/노트 클릭)
      ▼
[2] 노트북→노트 리스트   ← 계층 탐색 (좌 사이드바 2단 트리, status 인라인 토글, 칩)
      │  (노트 클릭 / 새 노트)
      ▼
[3] 노트 상세/작성  ★ 주인공 — A의 심장 (body.popup.type2)   ← 7장 상세
      │                              │
      ▼ (참조의 '계보' 보기)          ▼ (모듈 링크 → formulator, ELN 밖·목업)
[4] 계보(lineage) 뷰   ◆ D2 — 파생 트리, status 색상

[보조/후순위]
[5] 검색·회상 (C) — 전역 검색 + 노트 열면 관련 노트 자동 노출
[6] 최근 사용 태그 모아보기 — 태그 재사용 유도
```

| 화면 | 드러내는 심장 | 우선순위 |
|---|---|---|
| [1] 개인 홈 | B (정리) — 차별 첫인상 | 조연(감탄 포인트) |
| [2] 리스트 | 계층 + status 운영 | 조연 |
| **[3] 노트 상세** | **A (기록) + 참조 진입** | **주인공** |
| [4] 계보 뷰 | D (연결) — 독창성 임팩트 | 조연(감탄 포인트) |
| [5][6] 검색·태그 | C — 부산물 | 후순위 |

**데모 동선:** 홈[1] → 노트북 리스트[2] → 노트 작성[3] → 참조에서 계보[4]. 한 줄로 이어진다.

## 7. 주인공 [3] 노트 상세/작성 — 레이아웃

### 7.1 구조
```
┌──────────────────────────────────────────────────────────┐
│ 헤더: [제목 입력]   [status 탭토글: 진행중·성공·실패·보류] │
│        카테고리: 배합실험 ▾                                │
├────────────┬─────────────────────────────┬───────────────┤
│ 좌: 섹션    │ 중앙: 노트 본문(카드 스택)   │ 우: 연계패널  │
│ 메뉴        │  · 카테고리 골격 섹션들      │ (접이식)      │
│ (배합표·    │  · 배합실험: 배합표·물성     │  · 참조 원료  │
│  관찰·결론) │    임베드(formulator 목업)   │  · 참조 배합  │
│            │  · 텍스트·표·차트 블록       │  · 프로젝트   │
│            │                             │  · [[노트링크]]│
│            │                             │  · 계보 미니맵 │
├────────────┴─────────────────────────────┴───────────────┤
│ 하단: 자유태그 [ #점도이슈  #고객A  + ]                    │
│        공유·코멘트 (노트북 상속, 접이식)                   │
└──────────────────────────────────────────────────────────┘
```

### 7.2 우측 연계패널 기본 상태 = status 연동
- **진행중** → 패널 **접힘**: 깨끗한 쓰기 캔버스(A 강조, "안 귀찮네"가 먼저).
- **결말(성공/실패/보류)** → 패널 **펼침**: 연결·계보가 드러남(D 강조, "살아있네"가 먼저).
- 연구원이 수동 토글로 언제든 덮어쓸 수 있음.
- **구현 비용:** status 값에 따라 우측 패널 `class` 토글(`.collapsed`) 한 줄 수준 — 영리하면서 싸게.

### 7.3 무첨부(attachment-free) 철학 유지
- 불투명 파일 업로드 대신 네이티브 구조화 데이터(입력필드·표·Highcharts 차트·레지스트리 구조식·타모듈 라이브링크)로 실험이 완성됨. 문서/파일은 SharePoint 강력 연동으로 처리(별도, 이번 MVP 화면 범위 밖).

## 8. 범위 밖 / 후순위 (이번 MVP 비포함)

- **C 검색·회상** — D 그래프 생기면 따라옴. [5][6]은 목업 수준만.
- **E 공유 확장** — 노트북 단위 최소만. 동시편집·세밀 권한은 나중.
- **LLM 개념 추출 태깅** — 나중에 켜는 옵션(저장 시 1회, Haiku급). MVP는 LLM 0.
- **새 축 F/G/H/I** — F(실패 자산화)는 status로 흡수됨. G(맥락 자동 초안)·H(질문 추적)·I(회고/통찰, recap·wrap-up 재미 요소)는 보관.
- **합성/분석 카테고리 고유 골격** — 값만 열어두고 골격은 점진 확장.
- **결재(승인 워크플로우)** — MVP 제외.

## 9. 미해결/구현 시 결정할 디테일 (writing-plans로 이월)

- 배합실험 카테고리의 **골격 섹션 구성** 세부(어떤 섹션이 기본으로 깔리나).
- [4] 계보 뷰의 **시각화 형태**(트리 / 타임라인 / 노드그래프 중 무엇).
- [1] 개인 홈의 **위젯 구성**(어떤 카드가 자동 배치되나).
- formulator/Rawmathub **임베드 카드의 목업 모양**.
